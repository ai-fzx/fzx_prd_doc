# 需求书写规范

> 本文档定义 PRD 中各类内容的书写标准，确保需求清晰、可验收、无歧义。

---

## 一、用户故事规范

### 标准格式

```
作为 [角色]，我希望 [能力/行为]，以便 [价值/目标]
```

### 规则

1. **一个故事 = 一个可交付能力**：如果包含「和」「以及」，拆成多条
2. **角色必须具体**：避免「用户」，用「项目经理」「医生」「电商买家」
3. **行为必须可测试**：说清楚做什么，不说怎么做（实现细节留给工程师）
4. **价值必须业务相关**：说清楚对业务/用户的意义，不是技术收益

### 正确示例

```
✅ 作为 [在线购物的买家]，我希望在下单后 30 分钟内收到确认邮件，
   以便我确认订单信息正确并安心等待发货

❌ 作为 [用户]，我希望系统好用（优化体验）
```

### 错误对照

| 错误写法 | 问题 | 正确写法 |
|---------|------|---------|
| 作为用户，优化体验 | 角色模糊，行为不可测 | 作为付费会员，我希望加载时间 < 2s，以便快速浏览商品 |
| 提升转化率 | 行为缺失 | 作为运营，我希望看到商品页跳出率，以便优化选品 |
| 和/以及连接多个动作 | 一个故事多个交付 | 拆成 R1、R2 |

## 二、验收标准规范

### 标准格式（三段式）

```
Given [前置条件]
When [触发操作]
Then [预期结果]
```

### 简化格式

```
当 [触发事件] 时，系统/用户应 [预期行为]
```

### 规则

1. **可量化**：包含数字、比率、状态变更等可测量的结果
2. **可重复**：同一条件每次都能验证通过
3. **有明确责任方**：系统行为 vs 用户行为要分清
4. **覆盖正常和异常**：正常路径 + 边界/错误处理

### 正确示例

```
✅ R1：当买家完成下单支付时，系统应在 30 分钟内发送确认邮件到买家
    注册邮箱，邮件包含订单号、商品列表和预计发货时间

✅ R2：当用户在 24 小时内未登录，系统应发送最多 1 条提醒邮件，
    且邮件包含退订链接

❌ R1：提升用户体验（无法验证）
❌ R2：尽快发送提醒（时间不明确）
```

## 三、禁止表述

| 禁止写法 | 问题 | 正确写法 |
|---------|------|---------|
| 优化体验 | 不可衡量 | 首屏加载 < 2s，错误率 < 0.1% |
| 提升性能 | 不可衡量 | API 响应时间 P99 < 200ms |
| 可能/尽量/视情况 | 模糊不确定 | 明确规则，或列出具体例外情况 |
| 友好/易用/简洁 | 主观感受 | 具体交互步骤数，或字段填写数量 |
| 必要时 | 触发条件不清 | 明确条件（如：连续失败 3 次时） |
| 支持/支持多种 | 数量不明 | 明确数量（至少 3 种，最多 10 种） |

## 四、优先级定义

| 优先级 | 定义 | 说明 |
|--------|------|------|
| **P0** | 必须交付，阻塞发布 | 核心功能缺失 = 产品无法上线 |
| **P1** | 重要功能，本期完成 | 重要但有 workaround |
| **P2** | 优化增强，下期考虑 | 锦上添花，可延迟 |

### 判断方法

- 问自己：**没有这个功能，产品能上线吗？**
  - 能上线但体验差 → P1
  - 不能上线 → P0
  - 有则更好，没有也行 → P2

## 五、范围边界写法

### In Scope

- 具体说「做什么」，不只说功能名
- 包含适用条件（如：支持的平台、版本）

### Out of Scope

- 明确说「不做什么」，减少争议
- 避免「暂不」「以后再说」—— 用「本期不做」

### 示例

```
- **In scope**：
  - 支持 Chrome、Firefox、Safari 最新两个版本
  - 支持 1920×1080 和 1366×768 两种分辨率

- **Out of scope**：
  - 移动端适配（由独立 H5 项目负责）
  -  IE 浏览器兼容
```

## 六、指标定义

所有指标需包含：

1. **名称**：清晰的指标名
2. **定义**：怎么计算（分子/分母）
3. **基线**：当前值是多少
4. **目标**：期望达到的值
5. **测量方式**：从哪个系统/日志提取数据

### 示例

```
成功指标：
- 提醒送达率 = 成功发送提醒数 / 触发提醒总数
- 基线：30%（当前值）
- 目标：≥ 99%（提升 69pp）
- 数据来源：提醒系统日志
```

## 七、排版规范

- 标题层级：一级 `#` 用于文档标题，二级 `##` 用于章节
- 表格优先：列表信息用表格，更清晰（需求列表、指标定义）
- 编号一致：需求 ID 统一用 R1、R2…，与验收标准对应
- 占位符格式：`[请补充：xxx]`、`[待确认]`、`[数据缺失]`
- Markdown 输出：可直接复制到 Confluence/飞书/Notion
