# PRD 需求书写规范

## 用户故事写法

### 标准格式
> **作为** [角色]，**我希望** [能力]，**以便** [价值]

### 好例子
- 「作为注册用户，我希望能通过邮箱找回密码，以便在忘记密码时自助恢复账户」
- 「作为运营人员，我希望导出用户行为报表，以便分析转化漏斗」

### 坏例子（不可验收）
- 「优化找回密码体验」→ 改为：用户可在 5 分钟内收到重置链接
- 「提升报表加载速度」→ 改为：报表在 3 秒内加载完成

---

## 验收标准写法

### Given-When-Then 格式
```
Given（前置条件）：用户已完成 [某操作]
When（触发）：用户执行 [某动作]
Then（预期）：系统应 [某行为]
```

### 检查清单格式
每条需求至少一条可验证条件：
- R1：当用户提交已验证邮箱时，系统在 5 分钟内发送重置链接；链接 24h 有效
- R2：当用户超过 3 次输入错误密码时，账号锁定 30 分钟

## 优先级定义

| 优先级 | 定义 | 团队约定 |
|--------|------|----------|
| P0 | 必须本期交付，阻塞发布 | |
| P1 | 重要但可下期，不阻塞 | |
| P2 | 可选、锦上添花 | |

## 常见错误

### 1. 需求不可验收
```
错误：提升用户体验
正确：用户操作到响应 < 2s，操作成功率达 99.5%
```

### 2. 需求过于模糊
```
错误：优化搜索功能
正确：搜索结果召回率 > 95%，TOP 3 结果相关性 > 90%
```

### 3. 范围蔓延
在「非目标」节明确列出本期不做的事项，避免评审时 scope creep。

### 4. 缺少成功指标
没有指标的 Goal = 愿望，不是目标。至少写一个可量化的指标及基线。
