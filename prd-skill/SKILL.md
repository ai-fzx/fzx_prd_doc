# PRD 产品需求文档技能

## 执行前

**输入范围**：用户提供的产品背景、目标、用户与场景，或已有的 PRD 草稿/片段。

若用户只说「写一份 PRD」「写需求文档」等**未提供任何要点**，先询问：
- 产品/功能名称
- 要解决的问题
- 目标用户
- 核心目标或成功指标

**前提确认**：有至少「产品/功能主题与目标」后，再按下方模板输出；缺项用 `[请补充：xxx]` 占位。

---

## 文档结构（必填节）

按以下顺序输出，每节必须有内容或占位说明：

### 1. 背景与问题
要解决什么问题、为谁、当前痛点或机会；可 1～3 段。

### 2. 目标与成功指标
- **目标**：[业务/产品目标]
- **成功指标**：可衡量指标（如转化率、留存、DAU、NPS）；若无数据写 `[请补充指标与基线]`

### 3. 用户与场景
- **目标用户**：[具体用户角色]
- **核心场景**：1）… 2）… 3）…

### 4. 功能需求
每条带 **ID**（如 R1、R2）与 **优先级**（P0/P1/P2）

| ID | 描述（用户故事或需求） | 优先级 |
|----|----------------------|--------|
| R1 | 作为 [角色]，我希望 [能力]，以便 [价值] | P0 |
| R2 | [请补充] | P1 |

### 5. 范围与边界
- **In scope（本期做）**：
- **Out of scope（本期不做）**：

### 6. 验收标准
每条需求对应可验证的验收条件，格式如「R1：当…时，应…」
- R1：[验收条件]
- R2：[请补充]

### 7. 依赖与约束
关键依赖、上线时间、合规或性能约束；无则写「无」或 `[请补充]`

---

## 文档结构（可选节）

### 非目标
本期明确不解决的问题，避免范围蔓延。

### 附录
术语表、参考链接、变更记录。

---

## 需求书写规范

### 用户故事格式
> **作为** [角色]，**我希望** [能力/行为]，**以便** [价值/目标]

一条故事只描述一个可交付能力，避免「和」「以及」拆成多条。

### 验收标准格式
> **Given** 前置条件 **When** 操作 **Then** 预期结果

或「当…时，应…」

### 禁止表述
| 禁止 | 改为 |
|------|------|
| 优化体验 | 首屏加载 < 2s，错误率 < 0.1% |
| 提升性能 | API 响应 < 200ms |
| 可能/尽量/视情况 | 明确规则或列出例外 |

### 优先级定义
- **P0**：必须本期交付，阻塞发布
- **P1**：重要但可下期
- **P2**：可选、锦上添花

---

## 评审模式

若用户提供**已有 PRD** 并要求「检查」「评审」，输出：

- ✅ **符合项**：已含必填节且内容完整
- 🔴 **必改**：缺失节或不可验收需求（必须补充才能发布）
- 🟡 **建议**：可优化但不影响发布
- ⚪ **可选**：锦上添花的优化

---

## 输出格式

- 输出为完整 **Markdown** 文档，可直接复制到 Confluence/Notion/飞书
- 标题层级：一级 = 文档标题，二级 = 各节
- 占位符统一为 `[请补充：xxx]` 或 `[待确认]`
- 文档开头写 `# PRD：[产品/功能名称]`
